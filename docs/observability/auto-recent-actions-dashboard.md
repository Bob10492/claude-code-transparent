# Auto-refresh recent action ids in the dashboard

## Why this exists

`semantic_viewer_app.html` is a static HTML file. A browser page opened from disk cannot directly query `.observability/observability_v1.duckdb` every time it loads, so the latest action ids must be written into the HTML when the viewer is generated.

The one-command generator below keeps this automatic: whenever you generate a semantic viewer, it also refreshes the dashboard's Recent User Actions card.

## Use this command instead of running two commands

Latest action:

```bash
bun run scripts/observability/deep_explain_action_with_recent_dashboard.ts --latest
```

Specific action:

```bash
bun run scripts/observability/deep_explain_action_with_recent_dashboard.ts --user-action-id <user_action_id>
```

Then open:

```text
ObservrityTask/action-reports/deep/semantic_viewer_app.html
```

The dashboard will include the latest five indexed action ids as read-only copy fields.

## What the wrapper does

It runs:

```bash
bun run scripts/observability/deep_explain_action.ts <your args>
bun run scripts/observability/patch_semantic_viewer_recent_actions.ts
```

So you do not need to remember the second patch command.

## Can this be fully automatic on browser refresh?

Not with the current static-file design. To make it update on every browser refresh without running any command, the dashboard would need one of these larger changes:

1. a small local HTTP server/API that reads DuckDB and serves recent actions dynamically;
2. a build step that embeds recent actions into the generated HTML, which is what this wrapper automates;
3. a database-export JSON that is regenerated by the observability collector itself.

For the current repository, the safest implementation is option 2: auto-refresh during viewer generation.
